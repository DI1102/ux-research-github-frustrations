# UX Research: GitHub Open Source Contribution Experience

## Overview

This UX research project explores the challenges faced by beginner developers when making their first contribution to GitHub open-source projects.

The research focused on understanding common frustrations, confusing workflows, emotional barriers, and learning behaviors among first-year BTech CSE students who are new to GitHub and open-source collaboration.

---

## Research Goal

To identify usability issues that prevent beginners from confidently contributing to open-source repositories and to discover opportunities for improving the onboarding experience.

### Research Questions

* What frustrates beginners while contributing to GitHub projects?
* Which GitHub features are most confusing?
* How do users currently overcome these challenges?
* What improvements would help first-time contributors succeed?

## Research Method

### User Interviews

Semi-structured interviews were conducted with participants.

Sample Questions:

1. What are you trying to accomplish on GitHub?
2. Which part of GitHub confuses you the most?
3. Have you ever abandoned a contribution attempt?
4. What do you do when you encounter an error?
5. What would make GitHub easier for beginners?

---

## Raw Research Findings

### Common Observations

* Users struggled to understand the difference between Fork, Clone, and Branch.
* Most participants watched YouTube tutorials before taking action.
* README files were often outdated.
* CI failure icons were ignored because users did not understand them.
* Merge conflicts caused anxiety and confusion.
* Users blamed themselves instead of the platform.

### Example Quotes

> "Fork and clone look the same to me."

> "I watched four videos before making my first contribution."

> "The merge conflict looked like file corruption."

> "I felt stupid for an hour."

> "I didn't know what CI failure meant."

# Affinity Mapping

After collecting interview responses, similar observations were grouped together to identify patterns.

## 1. Fork / Clone / Branch Confusion

### Findings

* Users could not differentiate Git concepts.
* No visual workflow existed.

### Insight

Beginners need a visual contribution workflow before interacting with repositories.

---

## 2. Misleading "Good First Issue" Labels

### Findings

* Issues lacked context.
* Missing setup instructions.
* No clear file paths.

### Insight

Labels create expectations that are not fulfilled by issue descriptions.

---

## 3. Merge Conflict Anxiety

### Findings

* Conflict markers looked broken.
* Users deleted branches instead of fixing issues.

### Insight

Fear of making mistakes prevents users from learning debugging skills.

---

## 4. Outdated Documentation

### Findings

* README instructions were old.
* Setup steps failed.

### Insight

Outdated documentation reduces trust and increases frustration.

---

## 5. GitHub Actions & CI Confusion

### Findings

* Users ignored CI failures.
* Technical terminology was unclear.

### Insight

Beginners need plain-language explanations instead of technical jargon.

---

# Key Research Insights

### Insight 1

All participants relied on external help before making a contribution.

### Insight 2

Users feared breaking the repository more than making coding mistakes.

### Insight 3

Documentation quality directly influenced confidence levels.

### Insight 4

Most failures were caused by workflow confusion rather than coding ability.

### Insight 5

Beginners require guidance at the exact moment they encounter errors.

---

# Opportunity Areas

Based on the research, several opportunities were identified:

### Visual Contribution Flow

Provide a Fork → Clone → Branch → Pull Request diagram.

### Beginner-Friendly Issue Templates

Include setup instructions, file locations, and expected outputs.

### README Freshness Indicator

Flag outdated documentation automatically.

### Merge Conflict Assistance

Provide inline explanations and guided conflict resolution.

### Human-Friendly CI Feedback

Convert technical errors into plain-language explanations.

---

# Conclusion

The research revealed that beginner developers struggle more with understanding GitHub workflows than with writing code itself.

The biggest barriers were confusing terminology, lack of guidance, outdated documentation, and fear of making mistakes.

By introducing visual workflows, beginner-friendly explanations, and contextual guidance, GitHub could significantly improve the onboarding experience for new open-source contributors.
